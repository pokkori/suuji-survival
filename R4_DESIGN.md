# R4設計書: 数字サバイバル

## Part 1: 根本原因の徹底分析

### R1→R2→R3 実装履歴

| ラウンド | 実装内容 | 本質的変化 |
|---|---|---|
| R1 | 6x10グリッド、スワイプで合計10消去、コンボ/フィーバー、4種特殊ブロック、25実績、デイリー、ランキング、ショップ、テーマ | ゲームの骨格完成 |
| R2 | Reanimatedアニメーション、ハプティクス、チュートリアル、Wordle式シェア、消去パーティクル、画面シェイク、フィーバー虹背景 | 視覚エフェクト追加 |
| R3 | 衝撃波エフェクト、Web Audio SE、ハプティクス修正、5+コンボ白フラッシュ | さらにエフェクト追加 |

### なぜ実装しても点数が上がらない（32→27）のか: 3つの根本原因

**根本原因1: ゲームループに「受動的快感」がゼロ**

ぷよぷよの核心は「自分が消した後、重力で落ちてきた結果として"勝手に"連鎖が起きる」こと。プレイヤーは仕掛けを作り、あとは見ているだけで気持ちよくなれる。

数字サバイバルの現状コード（`useGameEngine.ts` L260-365 `handleSwipeEnd`）:
1. プレイヤーがスワイプして合計10を作る
2. `clearCells` → `applyGravity` → **終了**
3. 重力後に新たなマッチが発生するかのチェックが**一切ない**

つまり「消す→落ちる→また消える」というカスケード連鎖が構造的に不可能。R2/R3でパーティクルや衝撃波を追加しても、「1回消したら終わり」という平坦な体験は何も変わっていない。エフェクトの派手さではなく、ゲームメカニクスの深さが欠落している。

**根本原因2: 戦略性（消し順を考える動機）がない**

Threes!やDrop7では「今この手を打つと、次にこうなるから...」という先読みが楽しさの中核。数字サバイバルでは:
- 合計10になる隣接セルを見つけてスワイプ → 消える → 上から新しい行が降ってくる
- 降ってくる行はランダムで、プレイヤーが配置を制御できない
- 消した後の盤面がどうなるかを「計算して仕掛ける」要素がゼロ

現状の「コンボ」は2秒以内に連続で消すだけの時間制約であり、連鎖を仕掛ける頭脳的快感とは無関係。

**根本原因3: R2/R3の改善が「表面装飾」に終始し、ゲーム体験の本質に触れていない**

R2/R3で追加されたもの:
- パーティクル、衝撃波、画面シェイク、白フラッシュ、SE、ハプティクス

これらは全て「消去した瞬間の1フレームの装飾」であり、ゲームの「面白さの構造」（=連鎖・戦略・先読み）には一切影響しない。むしろエフェクト過多で「ゲームの浅さ」が際立ち、評価者に「飾りだけ」と判断されてスコアが下がった。

### ぷよぷよ・Threes!との決定的差異（3点）

| 比較軸 | ぷよぷよ / Threes! | 数字サバイバル（現状） |
|---|---|---|
| **1. カスケード連鎖** | 消去→重力→自動再マッチ→さらに消去（受動的快感の源泉） | 消去→重力→終了。再マッチなし。1手で全てが完結 |
| **2. 仕掛ける楽しさ** | 3手先を読んで配置し、大連鎖を狙う知的興奮 | 「今見えている10の組み合わせ」を見つけるだけの反射ゲーム |
| **3. テンポの緩急** | 仕掛け中（緊張）→連鎖発動（解放）の感情曲線 | 常に同じテンション。消しても消さなくても感情が平坦 |

## Part 2: R4改善設計（コードで解決できる問題のみ）

### 改善1: カスケード連鎖システム（最重要・楽しい度の根幹）

- **対象ファイル**:
  - `/c/Temp/number-survivor/engine/cascadeLogic.ts`（新規作成）
  - `/c/Temp/number-survivor/hooks/useGameEngine.ts`
  - `/c/Temp/number-survivor/types/index.ts`

- **変更内容**:

  **1. 自動マッチ検出エンジン（`cascadeLogic.ts` 新規）**:
  ```
  findAutoMatches(grid: Grid): Match[]
  ```
  - 重力適用後のグリッドをスキャンし、隣接セルの合計が10になるグループを自動検出
  - 検出アルゴリズム: 各セルから開始し、BFS/DFSで隣接セルを探索、合計10になる組み合わせを貪欲に検出
  - 優先順: セル数が多い組み合わせを優先（3セル以上の連鎖を優先マッチ）
  - 連鎖段数をカウント（1段目=手動消去、2段目=最初の自動マッチ、3段目=...）

  **2. カスケードフェーズの追加（`useGameEngine.ts`）**:

  `handleSwipeEnd` を以下のフローに変更:
  ```
  手動消去 → clearCells → applyGravity
    → findAutoMatches → マッチあり？
      → YES: 300msウェイト（演出用）→ 自動消去 → applyGravity → findAutoMatches → ループ
      → NO: カスケード終了、通常状態に復帰
  ```

  - カスケード中はphaseを `'cascading'` に設定（プレイヤー入力を受け付けない）
  - 各段に300msのディレイを入れ、連鎖が視覚的に「段階的に」起きるようにする
  - 連鎖段数ごとにスコア倍率を乗算（2段目=x1.5、3段目=x2.0、4段目=x3.0、5段目以上=x4.0）

  **3. 連鎖演出**:
  - 連鎖段数をComboDisplayとは別に「連鎖！」「2連鎖！！」「3連鎖！！！」と画面中央に大きく表示
  - 各段の自動消去時にもパーティクル・SE・画面シェイクを発火
  - 3連鎖以上でスロー演出（300ms→500msにディレイ拡大）で「すごいことが起きている」感を演出

- **ユーザー行動の変化**:
  「この5を消せば、上の3と7が隣接して自動で消えて、さらに...」と先読みするようになる。消す瞬間ではなく「消した後に何が起きるか」にワクワクする体験に変わる。ぷよぷよの「仕掛けて見守る」快感が生まれる。

- **影響軸と点数変化**:
  - 楽しい度: 5→8（連鎖の受動的快感がゲームの中核体験になる）
  - 表現性: 5→7（連鎖演出が視覚的インパクトを構造的に生む）
  - 差別化: 4→6（「合計10×カスケード連鎖」は類似ゲームにない組み合わせ）

### 改善2: 連鎖誘発グリッド生成（戦略性の担保）

- **対象ファイル**:
  - `/c/Temp/number-survivor/engine/dropLogic.ts`

- **変更内容**:

  現状の `generateNewRows` は完全ランダムだが、これでは連鎖が偶然にしか発生しない。以下の調整を加える:

  **補数ペア挿入ロジック**:
  - 新しい行を生成する際、20-30%の確率で「補数ペア」（合計10になる2つの数字、例: 3と7、4と6）を隣接配置する
  - これにより、重力で落下した後に自動マッチが発生しやすくなる
  - 難易度が上がるにつれて補数ペア率を15%→10%に下げる（上級は自力で見つける必要あり）

  **実装詳細**:
  ```typescript
  // generateNewRows内で、一定確率で隣接する2セルを補数ペアにする
  const COMPLEMENT_PAIR_CHANCE = 0.25; // 25%の確率で補数ペア挿入
  const pairs: [NumberValue, NumberValue][] = [[1,9],[2,8],[3,7],[4,6],[5,5]];
  ```

- **ユーザー行動の変化**:
  「あ、7と3が上に来る...今この列の4を消せば、7が落ちてきて3と合体する」という計画的プレイが可能になる。NextQueueのプレビューを見る意味が生まれる。

- **影響軸と点数変化**:
  - 楽しい度: +1加算（連鎖が「狙える」ので達成感が生まれる）
  - 差別化: +1加算（ランダムではなく「準備された偶然」が生まれる）

### 改善3: 連鎖スコアポップアップ＋累積表示

- **対象ファイル**:
  - `/c/Temp/number-survivor/components/ChainDisplay.tsx`（新規作成）
  - `/c/Temp/number-survivor/app/game.tsx`
  - `/c/Temp/number-survivor/engine/scoreCalc.ts`

- **変更内容**:

  **ChainDisplay コンポーネント**:
  - 連鎖発生時に画面中央に大きく表示: 「2連鎖! +150」「3連鎖!! +300」
  - フォントサイズを連鎖段数に応じて拡大（2連鎖=32pt、3連鎖=40pt、4連鎖以上=48pt）
  - 色を段数で変化: 2=オレンジ、3=赤、4=紫、5+=虹グラデーション
  - Reanimated: scaleで0→1.2→1.0のバウンスイン + 300ms後にフェードアウト
  - 各段のスコアが画面に残り、最後に合計が「TOTAL: +XXX」と表示されてから消える

  **scoreCalc.ts修正**:
  - `calculateChainScore(chainLevel: number, baseScore: number): number` を追加
  - 連鎖倍率: 1段目=x1, 2段目=x1.5, 3段目=x2.5, 4段目=x4, 5段目以上=x6
  - これにより3連鎖以上を狙うインセンティブが非常に大きくなる

- **ユーザー行動の変化**:
  連鎖が起きるたびにスコアが目に見えて跳ね上がるので、「もっと大きな連鎖を仕掛けたい」という欲求が生まれる。ハイスコア更新の鍵が「いかに大連鎖を組むか」になり、リプレイ動機が強まる。

- **影響軸と点数変化**:
  - 表現性: +1加算（連鎖ごとの段階的フィードバックでドラマ性が生まれる）
  - 楽しい度: +1加算（スコアの跳ね上がりが「やった！」感を倍増）

### 改善4: シェアテキスト圧縮＋連鎖ハイライト

- **対象ファイル**:
  - `/c/Temp/number-survivor/app/game.tsx`（`generateWordleGrid`、`handleShare`）

- **変更内容**:

  現状のシェアテキストは10行6列=60文字のグリッドで冗長（10行で画面を埋め尽くす）。以下に変更:

  **シェアフォーマット刷新**:
  ```
  数字サバイバル #42
  🔥 12,500pts | 最大3連鎖
  🟩🟩🟨🟥🟩🟩
  🟩🟨🟨🟥🟩🟨
  ```

- **影響軸と点数変化**:

### 改善5: カスケード中のスローモーション演出＋連鎖SE

- **対象ファイル**:
  - `/c/Temp/number-survivor/utils/sound.ts`
  - `/c/Temp/number-survivor/app/game.tsx`

- **変更内容**:

  **連鎖専用SE（`sound.ts`）**:
  - `playChainSound(chainLevel: number)`: 連鎖段数に応じてピッチが上がるSE
  - 1段目: C4(523Hz), 2段目: E4(659Hz), 3段目: G4(784Hz), 4段目: C5(1047Hz)
  - 5段目以上: ファンファーレ的なアルペジオ（C-E-G-C連続再生）

  **カスケード中の時間演出（`game.tsx`）**:
  - 3連鎖以上のカスケード中、セルの消去アニメーションをスローに（duration 250ms→500ms）
  - 連鎖が進むごとに背景が段階的に暗くなり、消去セルだけが光る「スポットライト演出」
  - 最終連鎖後に一瞬の静止（200ms）→ 全体フラッシュ → 通常速度に復帰

- **ユーザー行動の変化**:
  大連鎖の瞬間が「映画のクライマックスシーン」のように感じられる。スロー＋ピッチ上昇で「すごいことが起きている」感が体感で理解でき、思わず声が出る体験になる。

- **影響軸と点数変化**:
  - 表現性: 5→8（連鎖演出がプロゲームレベルの完成度に）
  - 楽しい度: 追加+1（スロー演出の映画的カタルシス）

## 実装優先順位

1. **改善1: カスケード連鎖システム** — これがないと他の全てが意味をなさない。ゲームの魂。
2. **改善2: 連鎖誘発グリッド生成** — 連鎖が偶然でなく「狙える」ようにする。改善1とセットで実装。
3. **改善5: スローモーション演出＋連鎖SE** — 連鎖の気持ちよさを10倍にする演出。改善1完成直後に実装。
4. **改善3: 連鎖スコアポップアップ** — スコアの視覚フィードバックで達成感を倍増。
5. **改善4: シェアテキスト圧縮** — 最後に対応。

## 補足: コードで解決できない問題（正直に申告）

- **収益性 2/10**: App Store / Google Play公開 + IAP/広告SDK統合が必要。コード実装は可能だが、開発者アカウント・審査が前提。
- **SEO・集客 1/10**: Expo Goでのローカル実行のみ。WebビルドしてVercelにデプロイするか、ストア公開が必要。
- **BGM未実装**: 長時間プレイの没入感にBGMは重要だが、音楽アセットの作成/調達が必要。Web Audio APIでプロシージャル生成は可能だが品質に限界がある。
